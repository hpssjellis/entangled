7 # cIRI
8 
9 A low level implementation of an IOTA node in C.
10 
11 This is a WIP [IOTA](https://iota.org/) node. It allows users to become part of the [IOTA](https://iota.org/) network as both a transaction relay and network information provider through the easy-to-use [API](https://iota.readme.io/reference).
12 
13 ## How to get started
14 
15 The IOTA network is an independent peer-to-peer network with a first-user, friend-to-friend, network structure:
16 
17 - As a 'first-user' network, to access the data streams and APIs that other users provide, you must first exchange your IP and port configuration with a current user.
18 
19 - As a 'friend-to-friend' network, you have the privilege of joining new users into the network through your node by adding them to your approved neighbors list â€” ensuring that you both broadcast to them and also receive their broadcasts.
20 
21 You can find neighbors quickly at both our [Discord Community](https://discord.gg/7Gu2mG5) and [Forum](https://forum.iota.org/).
22 
23 Everyone will be welcoming and very happy to help you get connected. If you want to get tokens for your testcase, please just ask in one of the communication channels.

25 ## Build and run cIRI
26 
27 cIRI is built and run through [bazel](https://www.bazel.build/).
28 
29 Clone the project
30 
31 ```
32 $ git clone https://github.com/iotaledger/entangled.git
33 $ cd entangled
34 ```
35 
36 *First build can take some time due to dependencies downloading.*
37 
38 ### For a mainnet node
39 
40 Create the databases, only the first time:
41 ```
42 $ sqlite3 ciri/db/tangle-mainnet.db < common/storage/sql/tangle-schema.sql
43 $ sqlite3 ciri/db/spent-addresses-mainnet.db < common/storage/sql/spent-addresses-schema.sql
44 ```
45 
46 Build and run cIRI
47 ```
48 $ bazel run -c opt --define network=mainnet -- ciri # optional flags
49 ```

65 ## Configuration
66 
67 ### Configuration file
68 
69 cIRI configuration file is a [YAML](http://yaml.org/) file with path `ciri/conf.yml`.
70 
71 An example file `ciri/conf.example.yml` is provided; you can rename it, uncomment and change values of options you want to use.
72 
73 Example syntax:
74 
75 ```yaml
76 log-level: debug
77 neighbors: "tcp://148.148.148.148:14265"
78 http-port: 14265
79 ```
80 
81 ### Command Line Interface
82 
83 Long option | Short option | Description | Example input
84 --- | --- | --- | ---
85 `--config` | `-c` | Path to the configuration file. | `--config ciri/conf.yml`
86 `--help` | `-h` | Displays the usage. |
87 `--log-level` | `-l` | Valid log levels: "debug", "info", "notice", "warning", "error", "critical", "alert" and "emergency". | `-l debug`
88 `--spent-addresses-db-path` | | Path to the spent addresses database file. | `--spent-addresses-db-path ciri/db/spent-addresses-mainnet.db`
89 `--tangle-db-path` | | Path to the tangle database file. | `--tangle-db-path ciri/db/tangle-mainnet.db`
90 `--tangle-db-revalidate` | | Reloads milestones, state of the ledger and transactions metadata from the tangle database. | `--tangle-db-revalidate false`
91 `--auto-tethering-enabled` | | Whether to accept new connections from unknown neighbors (which are not defined in the config and were not added via addNeighbors). | `--auto-tethering-enabled false`
92 `--max-neighbors` | | The maximum number of neighbors allowed to be connected. | `--max-neighbors 5`
93 `--mwm` | | Number of trailing ternary 0s that must appear at the end of a transaction hash. Difficulty can be described as 3^mwm. | `--mwm 14`
94 `--neighboring-address` | | The address to bind the TCP server socket to. | `--neighboring-address "0.0.0.0"`
95 `--neighboring-port` | `-t` | The TCP receiver port. | `--neighboring-port 1500`
96 `--neighbors` | `-n` | URIs of neighbouring nodes, separated by a space. | `-n "tcp://148.148.148.148:14265 tcp://[2001:db8:a0b:12f0::1]:14265"`
97 `--p-send-milestone` | | Probability of sending a milestone transaction when the node looks for a random transaction to send to a neighbor. Value must be in [0,1]. | `--p-send-milestone 0.02`
98 `--recent-seen-bytes-cache-size` | | The number of entries to keep in the network cache. | `--recent-seen-bytes-cache-size 1500`
99 `--reconnect-attempt-interval` | | The interval (in seconds) at which to reconnect to neighbors. | `--reconnect-attempt-interval 60`
100 `--requester-queue-size` | | Size of the transaction requester queue. | `--requester-queue-size 10000`
101 `--tips-cache-size` | | Size of the tips cache. Also bounds the number of tips returned by getTips API call. | `--tips-cache-size 5000`
102 `--http-port` | `-p` | HTTP API listen port. | `--http-port 14265`
103 `--max-find-transactions` | | The maximal number of transactions that may be returned by the 'findTransactions' API call. If the number of transactions found exceeds this number an error will be returned | `--max-find-transactions 100000`
104 `--max-get-trytes` | | Maximum number of transactions that will be returned by the 'getTrytes' API call. | `--max-get-trytes 10000`
105 `--remote-limit-api` | | Commands that should be ignored by API. | `--remote-limit-api "attachToTangle, addNeighbors"`
106 `--alpha` | | Randomness of the tip selection. Value must be in [0, inf] where 0 is most random and inf is most deterministic. | `--alpha 0.001`
107 `--below-max-depth` | | Maximum number of unconfirmed transactions that may be analysed to find the latest referenced milestone by the currently visited transaction during the random walk. | `--below-max-depth 20000`
108 `--coordinator-address` | | The address of the coordinator. | `--coordinator-address "EQS...VD9"`
109 `--coordinator-depth` | | The depth of the Merkle tree which in turn determines the number of leaves (private keys) that the coordinator can use to sign a message. | `--coordinator-depth 23`
110 `--coordinator-security-level` | | The security level used in coordinator signatures. | `--coordinator-security-level 2`
111 `--coordinator-signature-type` | | The signature type used in coordinator signatures. Valid types: "CURL_P27", "CURL_P81" and "KERL". | `--coordinator-signature-type KERL`
112 `--last-milestone` | | The index of the last milestone issued by the corrdinator before the last snapshot. | `--last-milestone 1050000`
113 `--max-depth` | | Limits how many milestones behind the current one the random walk can start. | `--max-depth 15`
114 `--snapshot-file` | | Path to the file that contains the state of the ledger at the last snapshot. | `--snapshot-file external/snapshot_mainnet/file/snapshot.txt`
115 `--snapshot-signature-depth` | | Depth of the snapshot signature. | `--snapshot-signature-depth 6`
116 `--snapshot-signature-file` | | Path to the file that contains a signature for the snapshot file. | `--snapshot-signature-file external/snapshot_sig_mainnet/file/snapshot.sig`
117 `--snapshot-signature-index` | | Index of the snapshot signature. | `--snapshot-signature-index 12`
118 `--snapshot-signature-pubkey` | | Public key of the snapshot signature. | `--snapshot-signature-pubkey "TTX...YAC"`
119 `--snapshot-signature-skip-validation` | | Skip validation of snapshot signature. Must be "true" or "false". | `--snapshot-signature-skip-validation false`
120 `--snapshot-timestamp` | | Epoch time of the last snapshot. | `--snapshot-timestamp 1554904800`
121 `--spent-addresses-files` | | List of whitespace separated files that contains spent addresses to be merged into the database. | `--spent-addresses-files "file0 file1"`
